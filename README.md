# FingerSimulator

2D 평면 링키지 에디터 + 기구학 시뮬레이터. 로봇 손가락 메커니즘 분석용 (Nature Communications 2021 소프트 핑거 논문 기반). 단일 HTML 파일 — 빌드 도구, 외부 의존성 없음.

## 기능

**에디터 모드**
- 탭(클릭)으로 링크(폴리라인 강체) 점 단위로 그리기 — 물리적으론 강체라서 그리는 순서는 힘 전달에 무관, 시각적 폴리라인일 뿐
- 점을 드래그해서 다른 링크에 스냅 → 조인트 생성
- **탭 선택 + 하단 액션바**: 빈 공간 탭 = 선택 해제 / 새 링크 시작. 기존 점 탭 = 선택. 링크 세그먼트 탭 = 링크 선택. 선택한 대상에 적용 가능한 동작(역할 설정 ground/input/friction, origin 지정, 삭제 등)이 하단 액션바에 버튼으로 뜸. 우클릭·컨텍스트 메뉴 없음 — 데스크톱/모바일 동일 모델
- 그리드 스냅, 포인트/링크 레이블, 레이블 크기 슬라이더
- 전체 undo 히스토리 (Ctrl+Z)
- 텍스트 패널: 메커니즘 전체를 텍스트로 직렬화/역직렬화 — 직접 편집 후 apply 가능. 복잡한 분기·추가 점은 텍스트에서 편집
- `quantize` 버튼: 모든 포인트 좌표를 정수로 반올림 (공유용 깔끔한 숫자)
- `fit` 버튼 + 휠/핀치 줌 + 두 손가락/미들클릭/space+드래그 팬 — 캔버스를 자유롭게 확대·이동
- **origin**: 점 선택 후 액션바 "set origin" 버튼으로 지정. 해당 점이 좌표계 원점이 되고 그리드 번호·텍스트 패널·커서 좌표가 모두 origin 기준 상대좌표로 표시됨 (기계 설계 시 "이 점 기준 몇 mm" 판단용)
- **자동 저장**: 모든 상태(링크·조인트·역할·origin·rest position·view·UI 설정)를 `localStorage`에 자동 저장. 새로고침·브라우저 재실행해도 복원됨

**모바일·터치 지원**
- Pointer Events 기반 — 마우스/터치/펜 단일 코드 경로
- 두 손가락 핀치 줌, 두 손가락 팬 (SVG `touch-action: none`으로 브라우저 기본 제스처 차단)
- 820px 이하 뷰포트에서 단일 컬럼 레이아웃. 텍스트 패널은 `JSON` 버튼으로 여닫는 풀스크린 드로어로 전환
- 핀치 제스처 시작 시 두 손가락 중점을 viewBox 앵커로 고정한 채 view.w/h 스케일 — 확대/축소 중 앵커 점이 화면에서 움직이지 않음

**시뮬레이션 모드**
- `capture` 버튼: 현재 포즈를 rest position으로 저장. `reset` 버튼으로 언제든 rest로 복원
- `input` 또는 `friction` 포인트 드래그 → 실시간 기구학 솔버 동작
- 올-오어-낫씽 프레임 롤백: 기구학적 데드존 진입 시 마지막 유효 프레임으로 즉시 복귀 (부분 이동 없음)
- 멀티 입력 지원: input 핸들 하나를 드래그하면 나머지 input 포인트는 메커니즘을 따라 자유롭게 이동

## 역할(Role) 종류

| 역할 | 표시 | 동작 |
|---|---|---|
| **ground** | 채워진 원 + 삼각형 + 해칭 | rest 위치에 고정. 2개 이상의 ground 포인트가 있는 바디는 완전 고정. 드래그 불가. |
| **input** | 파란 원 + 위쪽 화살표 | 기구학 드래그 핸들. 드래그 중 커서 위치에 핀 (weight 100). 조인트(1000)보다 약해서 도달 불가능한 포즈로는 끌고 가지 못함. |
| **friction** | 점선 원 | 기구학 드래그 핸들이면서 시뮬레이션 중 soft 저항. Weight 8 — 움직이지만 저항감 있음. |

## 솔버

`linkage-solver.js` — 2D 평면 링키지용 Levenberg-Marquardt 가중 최소제곱 솔버.

- 링크 = 3-DOF 강체 (x, y, angle). 포인트는 무게중심 기준 로컬 오프셋으로 저장.
- 모든 제약 = 가중 핀 방정식: `body.worldPt(p) == target` (핀당 2개 방정식)
- 조인트 = 두 바디의 공유 월드 포인트가 일치해야 한다는 핀 방정식 쌍
- `setFixed(body)`: 완전 고정 바디를 변수 벡터에서 제거

**가중치 설정**

| 제약 | Weight |
|---|---|
| 조인트 | 1000 |
| ground 핀 | 1000 |
| input 핀 (드래그 중) | 100 |
| friction 핀 | 8 |

조인트와 ground는 "절대 깨지면 안 되는" 하드 제약 — 실제 기계에서 핀이 빠지는 건 파손이므로 1000으로 고정. input은 "도달하면 좋은" 목표라서 10배 낮은 100. 커서를 따라가려고 조인트를 벌리는 일이 구조적으로 불가능해짐. friction은 8로 약한 저항.

**범위 제한 (롤백 판단)**

솔버 residual이 아닌 물리적 드리프트(SVG 픽셀 단위)로 판단:
- Ground drift: ground 앵커 포인트가 rest 위치에서 얼마나 이탈했는지
- Joint separation: 조인트 양쪽 월드 포인트 간 거리

서브스텝 중 `max(ground drift, joint separation) > 2.0 px` 이면 → 프레임 전체 롤백. 서브스텝 단위는 커서 이동 8px로 제한해 솔버가 항상 작은 증분 단계를 밟도록 함.

> residual을 쓰지 않는 이유: residual은 제약 개수·weight에 비례해 커지므로 의미 있는 임계값 설정이 불가능. 픽셀 단위 직접 측정이 메커니즘 규모와 무관하게 일관됨.

## 발견한 이슈 및 해결 과정

### 조인트에 ground + input 동시 설정 버그

**원인**: 역할이 ref 단위(`state.roles[ref]`)로 저장됨. 조인트는 서로 다른 링크의 두 ref를 연결하는 구조. 사용자가 거의 같은 위치를 두 번 우클릭할 때 picker가 다른 ref를 집어 서로 다른 역할이 쌓일 수 있었음.

**해결**: 역할을 조인트 그룹 단위로 관리. `getNodeRole(members)` / `setNodeRole(members, role)` API 도입 — 역할 설정 시 그룹 전체 ref를 먼저 초기화한 뒤 canonical rep에만 저장. 렌더러도 `getNodeRole(jointGroup(ref))`로 읽어서 물리적 핀 하나당 마커 하나만 표시.

부가 효과: 조인트 생성 시 ground/input 충돌 검증 — ground와 input 역할을 가진 두 포인트는 조인트로 연결할 수 없음.

### Rest config 저장 시 조인트 드리프트 고착 문제

**원인**: `saveRestConfig`가 각 ref의 현재 `link.points[local]`을 독립적으로 저장. 시뮬레이션 후 같은 조인트의 두 ref가 서로 다른 바디에 있으면 soft constraint 드리프트로 미세하게 다른 위치에 있을 수 있음 (weight 100은 충분히 뻑뻑하지만 무한하지 않음). 이 드리프트가 rest 스냅샷에 굳어져 다음 시뮬레이션이 불일치 상태에서 시작됨.

**해결**: 저장 전 조인트 그룹별 canonical position(정렬 기준 첫 번째 멤버 위치)을 계산해 모든 멤버를 해당 위치로 통일. Rest config 내 조인트 파트너들은 항상 정확히 일치함.

### 메커니즘 일부 이동 후 멈추는 "creep" 현상

**원인**: 롤백이 서브스텝 단위였음. 서브스텝 하나가 drift 체크에 실패하면 그 직전 서브스텝 위치에 머뭄. 사용자 입장에서는 메커니즘이 조금 움직인 뒤 멈춘 것처럼 보여 유효한 포즈로 오해하기 쉬웠음.

**해결**: 각 마우스 이벤트 시작 시점에 전체 프레임 상태(`link.points` 전체 + `kinSolver.bodies` 전체) 스냅샷. 어느 서브스텝이든 실패하면 프레임 전체 복원. 메커니즘은 목표 위치까지 완전히 이동하거나 전혀 이동하지 않음.

### 멀티 input 시 비드래그 input 포인트가 잠기는 문제

**원인**: `solveKinematics`가 매 프레임마다 모든 input 역할 포인트에 high-weight 핀을 추가. 드래그하지 않는 두 번째 input 포인트가 rest 위치에 고정되어 메커니즘 이동을 방해함.

**해결**: 현재 드래그 중인 ref에만 full input 핀(weight 100) 적용. 나머지 input 포인트는 핀 없이 조인트 제약만으로 자유롭게 따라오게 함.

### friction 역할이 텍스트 패널에 출력되지 않는 문제

**원인**: `parse()` 정규식이 `ground`와 `input`만 인식. friction은 직렬화는 정상이었으나 apply 시 조용히 무시됨.

**해결**: 정규식을 `/(ground|input|friction)/`으로 수정.

### 특정 포즈에서 메커니즘이 멈추고 조인트가 벌어지는 현상

**증상**: 어떤 포즈(특히 기구학적 한계 근처)에서 input을 드래그하면 메커니즘이 따라오지 않고, 그 자리에서 조인트가 3px 정도 벌어지면서 rollback 메시지(`stopped: joint L2.p2=L3.p1 sep 3.4px`)가 뜸. 다른 포즈에서는 정상 동작.

**디버그 과정**: 처음엔 ground 앵커 드리프트, `setFixed` 계산 오차, rest 포즈 저장 불일치를 의심했음. 콘솔에 바디 포즈 / 조인트 `wA - wB` 거리 / 솔버 residual을 찍어보니:
- **솔브 직전**: 모든 조인트 완벽히 닫혀 있음 (`wA == wB`, 거리 0px)
- **솔브 직후**: `L2.p2 ↔ L3.p1` 조인트가 3.06px 벌어짐, residual 약 450
- 즉, 솔버가 **의도적으로** 조인트를 벌린 것. 수치 오차가 아닌 우선순위 문제.

**원인**: 조인트 weight와 input weight가 둘 다 100이라서 최소제곱 cost가 동등. 기구학적으로 input이 도달 불가능한 포즈에서 솔버가:
- 조인트 닫으면 → input residual 50px × weight 100 = 5000/축 (거대한 cost)
- 조인트 3px 벌리면 → joint residual 3 × 100 = 300/축, input 가까워져서 residual 급감

수학적으로 "조인트를 살짝 벌려 input에 다가가는 해"가 더 낮은 cost가 됨. 솔버는 충실히 그 해를 반환. 그 결과 drift 체크에 걸려 rollback → 사용자는 "왜 안 움직이지?" 상태가 됨.

**해결**: 조인트와 ground weight를 1000으로 올려 input(100)보다 10배 강하게 만듦. 이제 수학적으로도 조인트를 벌리는 비용이 input을 포기하는 비용을 압도함. 
- 기구학적으로 가능한 포즈 → input 따라감 (조인트는 원래 닫혀 있으니 추가 비용 없음)
- 불가능한 포즈 → 조인트 유지한 채 가장 가까운 도달 가능 위치에서 멈춤. drift는 여전히 0이므로 rollback 안 걸리고, 사용자는 "더 이상 안 움직이네"만 체감.

**교훈**: 가중 최소제곱에서 "하드 제약"과 "소프트 목표"를 같은 weight로 두면, 솔버는 둘을 교환한다. 실제 기계의 우선순위(조인트 결속 >> 커서 추종)를 weight 비율에 반영해야 함.

### 인터랙션 모델 재설계: 우클릭 컨텍스트 메뉴 → 탭 선택 + 액션바

**배경**: 기존 UX는 우클릭 컨텍스트 메뉴로 역할 설정·origin 지정·삭제를 처리. 데스크톱에선 동작했지만 모바일/터치에선 우클릭 자체가 존재하지 않아 별도 코드 경로가 필요했음. 플랫폼별 분기를 늘리는 대신 "모든 입력 장치에서 동일하게 동작하는 하나의 모델"을 목표로 재설계.

**결정**:
- 우클릭 메뉴 완전 제거 (`contextmenu` 기본 동작만 막음)
- 탭 = 선택. 선택 상태(점/링크/드로잉 중)에 따라 하단 액션바에 적용 가능 버튼이 렌더링됨
- 빈 공간 탭 = 선택 해제 또는 새 링크 시작 (드로잉 상태에 따라)
- 드로잉 중엔 액션바에 "finish / cancel" 노출
- 액션바는 항상 자리를 차지(min-height 44px, placeholder 텍스트) — 레이아웃이 뛰지 않음

**부가 효과**: 같은 코드가 데스크톱 마우스 / 트랙패드 / 모바일 터치에서 모두 동작. 시각적 어포던스(선택 강조 + 버튼)가 명시적이라 신규 사용자도 발견하기 쉬움.

### 모바일 제스처: Pointer Events + 핀치 줌/팬

**배경**: SVG에 `mousedown/move/up`만 걸려 있어 터치는 브라우저 기본 제스처(스크롤·줌)로 해석됨. 드래그는 우연히 동작했으나 멀티 터치 제스처는 전혀 감지 못함.

**구현**:
- 모든 `mouse*` → `pointer*` 변환. `e.pointerType !== 'mouse'`로 호버/커서 로직 가드
- SVG에 `touch-action: none`으로 브라우저 기본 제스처 차단
- 캡처 단계에서 `activePointers: Map` 트래커 유지. 두 포인터가 잡히면 `gestureActive = true` 플래그 세팅, 기존 단일 포인터 핸들러는 이 플래그에서 바로 반환
- 제스처 시작 시점에 `pinchStart = { midVb, dist }`를 viewBox 좌표로 기록. `pointermove`에서 현재 거리비로 view.w/h를 스케일하고, 앵커(`midVb`)가 현재 화면 중점에 매핑되도록 view.x/y 재계산
- 제스처가 드래그 도중 시작되면 `onDragEnd(e)`로 드래그를 커밋한 뒤 제스처 모드 진입 — 드래그/제스처가 충돌하지 않음

**노트**: Chrome DevTools의 터치 시뮬레이션은 Shift+드래그로도 단일 터치만 합성함. 핀치 검증은 실기기 필요.

### 반응형 레이아웃 (≤820px)

**변경점**:
- `body` 그리드가 단일 컬럼으로 변경. 텍스트 패널(`side`)은 `position: fixed; inset: 0` 드로어로 전환, 툴바의 `JSON` 버튼으로 토글
